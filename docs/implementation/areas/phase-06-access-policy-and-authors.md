# Этап 06. Права доступа и авторство изменений

Что зафиксировано:

- права на просмотр, создание, редактирование, удаление, дублирование, перенос и загрузку файлов для базы знаний теперь идут через `Policy`, а не через разрозненные проверки в интерфейсе;
- `KnowledgeBasePresenter` синхронизирован с policy-слоем, поэтому UI и backend теперь опираются на одну и ту же логику доступа;
- для разделов и статей добавлены поля `created_by` и `updated_by`;
- при создании, редактировании, переносе и дублировании материалов автор изменений записывается в БД;
- сидеры перестроены так, чтобы база знаний получала автора-админа уже после создания пользователей;
- миграция `2026_04_23_000000_add_author_fields_to_knowledge_base_tables.php` применена.

Что это дает проекту:

- проще аудировать, кто создал или менял раздел/статью;
- проще строить будущие правила доступа на уровне владельца, редактора, администратора;
- структура базы лучше подготовлена для поиска, аналитики и AI-слоя поверх статей;
- backend стал устойчивее: права теперь не зависят только от фронта.

Что проверено:

- `vendor\bin\pint.bat --test`
- `php artisan test --filter=KnowledgeBaseArticleTest --stop-on-failure`
- `php artisan test --filter=KnowledgeBasePagesTest --stop-on-failure`

- определить, какие материалы сотрудник может только читать, а какие редактировать;
- после этого уже можно переходить к hardening: ограничения действий, более строгие проверки и UX для прав доступа.
