# Хранение контента базы знаний в БД

## Главный принцип

Самая ценная часть проекта после деплоя — не React-интерфейс, а структура данных.

Интерфейс можно переписать, перенести на другой фронт или адаптировать под React Native. Но статьи, файлы, изображения, видео, разделы и права доступа должны храниться так, чтобы их можно было безопасно читать, искать, переносить и анализировать.

## Где сейчас лежат статьи

Основная таблица:

- `knowledge_articles`

В ней хранится:

- раздел статьи: `knowledge_category_id`
- название: `title`
- адрес: `slug`
- иконка: `icon`
- краткое описание: `summary`
- обложка: `cover_url`
- plain-text версия статьи: `content`
- полный block-editor JSON: `blocks`
- публикация: `is_published`

Поле `blocks` остается главным источником для восстановления статьи в редакторе.

## Почему одного JSON недостаточно

JSON удобен для React-редактора, потому что он хранит статью как набор блоков.

Но только JSON неудобен для:

- просмотра данных в Adminer
- SQL-запросов
- поиска по конкретным блокам
- контроля файлов и изображений
- будущего AI-поиска по базе знаний
- миграции контента на другой интерфейс

Поэтому добавлен дополнительный SQL-слой.

## Нормализованные таблицы

### `knowledge_article_blocks`

В этой таблице каждая строка — отдельный блок статьи.

Например:

- абзац
- заголовок
- цитата
- список
- таблица
- код
- файл
- изображение
- видео
- ссылка

Ключевые поля:

- `knowledge_article_id` — к какой статье относится блок
- `block_uid` — стабильный ID блока из React-редактора
- `type` — тип блока
- `sort_order` — порядок блока в статье
- `plain_text` — текст блока для поиска и AI
- `html` — inline-форматирование, если оно есть
- `data` — технические данные блока

### `knowledge_article_assets`

В этой таблице каждая строка — отдельный файл, картинка или видео из статьи.

Ключевые поля:

- `knowledge_article_id` — к какой статье относится файл
- `knowledge_article_block_id` — к какому блоку относится файл
- `block_uid` — ID блока
- `kind` — `image`, `file`, `video`
- `url` — публичная ссылка
- `storage_path` — путь внутри Laravel storage
- `original_name` — имя файла
- `mime` — тип файла
- `size_label` — человекочитаемый размер
- `caption` — подпись

## Как это работает

При создании, сохранении или дублировании статьи Laravel:

1. сохраняет статью в `knowledge_articles`
2. сохраняет полный JSON в `blocks`
3. пересобирает SQL-слой блоков в `knowledge_article_blocks`
4. пересобирает SQL-слой файлов в `knowledge_article_assets`

Для ручной пересинхронизации есть команда:

```bash
php artisan knowledge-base:sync-article-structure
```

Она нужна, если мы массово изменили статьи, сидеры или формат блоков.

## Что смотреть в Adminer

В Adminer по базе `ds_art` сейчас важно смотреть:

- `knowledge_categories` — разделы базы знаний
- `knowledge_articles` — сами статьи
- `knowledge_article_blocks` — разложенные блоки статей
- `knowledge_article_assets` — файлы, изображения и видео из статей
- `users` — пользователи
- `employees` — сотрудники
- `employee_files` — файлы сотрудников

Если кажется, что база пустая, нужно проверить:

- выбрана ли именно база `ds_art`
- выполнены ли миграции
- выполнены ли сидеры
- нет ли просмотра другой базы или другого подключения

## Почему это важно для AI и поиска

Будущий AI-слой не должен парсить HTML из интерфейса.

Он должен работать с чистыми данными:

- название статьи
- раздел
- plain-text контент
- блоки
- файлы
- права доступа

Так поиск будет быстрее, точнее и безопаснее: модель не будет видеть скрытые статьи, если на этапе 6 мы правильно наложим права доступа.
