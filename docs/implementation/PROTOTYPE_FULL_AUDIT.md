# Полный аудит HTML-прототипа

## Зачем нужен этот файл

Этот документ фиксирует HTML-прототип как главный источник правды для MVP.

Здесь собраны:

- все крупные модули прототипа
- мелкие UI-фичи и микроинтеракции
- текущее состояние реализации в приложении
- список расхождений между прототипом и рабочим кодом

Главное правило:

- если в прототипе есть иконка, текст, кнопка, отступ, hover, меню, модалка, placeholder или отдельный сценарий, то в приложении это должно быть перенесено максимально близко, а не “примерно похоже”

## Прототип как source of truth

Для MVP прототип определяет:

- структуру экранов
- SPA-flow
- тексты интерфейса
- иконки
- состояния кнопок
- hover/focus/disabled/empty состояния
- размеры карточек и отступы
- порядок блоков на экранах
- mobile-поведение
- sidebar-дерево
- модалки и контекстные меню

Это значит:

- нельзя упрощать готовые сценарии без отдельного решения
- нельзя заменять существующий UX “на более удобный” только потому, что так проще в коде
- если есть расхождение, ориентируемся на прототип

## Базовая визуальная система прототипа

### Основа

- фон приложения: `#f5f4f1`
- surface: `#ffffff`
- secondary surface: `#f0efe9`
- основной текст: `#1a1a18`
- вторичный текст: `#6b6b67`
- третичный текст: `#9a9a95`
- accent: `#2d5a3d`
- accent light: `#eaf3ee`
- danger: `#c0392b`
- danger light: `#fdf0ee`

### Скругления

- базовые кнопки и поля: `10px`
- карточки и крупные контейнеры: `14px`

### Тени

- мягкая, легкая тень
- без тяжелого glossy-эффекта

### Шрифт

- `Onest`

### Важное правило по стилям

В рабочем приложении можно использовать те же значения напрямую.

То есть:

- цвета
- паддинги
- размеры шрифтов
- размеры карточек
- размеры кнопок

нужно переносить из прототипа буквально, а не приблизительно.

## Контроль разделами в прототипе

Это важный блок. Сейчас он в приложении еще не перенесен полностью.

### Что есть в прототипе у разделов в sidebar

У category-node в дереве есть:

- `+` кнопка добавить внутрь
- `···` кнопка действий
- hover-появление этих кнопок

### Что открывает меню раздела

Контекстное меню раздела содержит:

- `Переименовать`
- `Сменить иконку`
- `Добавить подраздел`
- `Удалить раздел`

### Как это работает в прототипе

- кнопки появляются на hover строки дерева
- меню открывается около конкретного раздела
- иконка меняется через emoji picker
- переименование идет отдельным сценарием
- удаление завязано на структуру дерева

### Состояние в приложении

Сейчас в приложении:

- дерево разделов есть
- переходы по разделам есть
- active/open state есть
- hover-кнопка `+` есть
- hover-кнопка `···` есть
- tree context menu есть
- `Переименовать` из дерева есть
- `Сменить иконку` из дерева есть
- `Добавить подраздел` из дерева есть
- `Удалить раздел` из дерева есть

Но пока нет полного паритета tree-controls:

- не доведен точный визуальный паритет popup-меню
- не перенесен drag/drop сценарий дерева и статей как в прототипе
- нет полного совпадения по spacing и поведению controls в collapsed/mobile режимах

### Вывод

## Правила переноса из прототипа

### 1. Тексты

Тексты кнопок, заголовков и подсказок берем из прототипа, если нет отдельного решения их менять.

### 2. Иконки

Если в прототипе для конкретного действия уже есть иконка, используем ее как ориентир.

### 3. Размеры

Размеры карточек, padding и border-radius не придумываем заново, а переносим из прототипа.

### 4. Состояния

Нужно переносить все состояния:

- default
- hover
- active
- disabled
- empty
- loading

### 5. Контекстные действия

Если действие в прототипе спрятано в hover-menu, не надо сразу выносить его в постоянную кнопку без причины.

### 6. Мелкие функции — не мелкие

В этом проекте “мелкие фишки” вроде:

- hover-кнопок
- локальных меню
- cover placeholder
- icon picker
- copy link
- dashed-card

это часть продуктового UX, а не косметика.

## Практический вывод на текущий момент

Прототип описывает не просто каркас, а почти полный продуктовый UX.

Поэтому дальше работа должна идти так:

1. Не только закрывать крупные модули.
2. Параллельно сверять экран с прототипом по мелким деталям.
3. Отмечать в статусе не просто “экран есть”, а “экран совпадает / не совпадает по UX”.
