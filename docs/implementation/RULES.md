# Правила реализации

## 1. Главный продуктовый ориентир

MVP в этом проекте определяется не списком “минимальных функций”, а текущим HTML-прототипом.

Правило:

- все пользовательские сценарии из `staff-cabinet (3).html` должны быть реализованы

## 2. Никакого лишнего стека

Используем только то, что решает конкретную текущую задачу.

На старте утверждено:

- Laravel 13
- PHP 8.4 через Laravel Herd
- MySQL
- Inertia.js
- React
- TypeScript
- Vite

Не добавляем автоматически:

- Redis
- Horizon
- Pulse
- Scout
- Meilisearch
- Tiptap
- глобальные state managers

Каждая новая технология должна отвечать на вопрос:

- какую конкретную текущую проблему она решает

## 3. Backend всегда главный по правам

Нельзя считать, что если кнопка скрыта на фронте, значит доступ закрыт.

Все критичные действия должны проверяться на backend:

- просмотр
- создание
- редактирование
- удаление
- перемещение
- изменение прав
- работа с файлами

## 4. HR-данные отделяем от employee view

Карточка администратора и карточка сотрудника не должны быть одной и той же проекцией данных.

Особенно это касается:

- заметок руководителя
- внутренних служебных комментариев
- будущих служебных атрибутов

## 5. Не делаем API ради API

По умолчанию:

- страницы идут через Laravel + Inertia
- JSON endpoints делаем только для действительно асинхронных действий

## 6. SPA-поведение — это правило всего приложения

Приложение должно ощущаться как SPA не только в одном модуле, а системно.

Правило:

- переходы по основным экранам идут через Inertia, без hard refresh как рабочего сценария
- create/update/delete действия должны сразу отражаться в интерфейсе после ответа сервера
- если пользователь видит изменения только после `F5`, значит flow реализован неправильно

Практический принцип:

- не городим кастомный SPA-runtime поверх Inertia без реальной необходимости
- сначала решаем проблему правильным mutation flow: локальная модалка, `useForm`, redirect в свежий Inertia page props, `preserveScroll`, partial reloads там, где это уместно
- только если Inertia-подхода реально не хватает, рассматриваем дополнительный клиентский слой

## 7. Не усложняем редактор заранее

Редактор должен покрыть функции HTML-прототипа.

Но:

- не нужно сразу тащить сложную редакторную платформу, если это еще не требуется
- Tiptap подключается только если простая реализация становится ненадежной или слишком дорогой в поддержке

## 8. Все спорные решения фиксируем

Если по ходу реализации принято важное решение, его нужно отразить в документации этой папки.

Особенно:

- что сознательно отложили
- что исключили
- что перенесли на post-MVP
