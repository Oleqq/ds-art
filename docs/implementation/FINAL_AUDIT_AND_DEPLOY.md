## Короткий вывод

Проект можно готовить к первому тестовому production-деплою. Основной риск сейчас не в коде приложения, а в инфраструктуре хостинга: Laravel-приложение нельзя безопасно выложить как обычный статический сайт, просто залив весь репозиторий в `public_html`.

Прямой ответ на текущий вопрос: просто залить файлы по FTP недостаточно. Это не обновит MySQL-схему и не создаст нужные столбцы. Для рабочего первого выката нужен либо импорт актуального SQL-дампа локальной базы, либо SSH-доступ для запуска миграций.

Лучший вариант для работы:

- попросить SSH/SFTP-доступ;
- попросить возможность выставить document root на папку `public`;
- попросить возможность запускать `php artisan migrate --force`, `php artisan storage:link`, `php artisan optimize`;
- попросить доступ к MySQL/phpMyAdmin и возможность делать SQL export/import;
- попросить права записи для `storage` и `bootstrap/cache`.

Если SSH не дадут, деплой через FTP возможен, но он будет ручным: файлы заливаются по FTP, база импортируется через phpMyAdmin, а `storage`/symlink нужно настраивать через панель хостинга или поддержку.

## FTP-only против GitHub + SSH

### Просто FTP

Плюсы:

- можно сделать уже сейчас, если есть только FTP и phpMyAdmin;
- не нужно настраивать CI/CD.

Минусы:

- нельзя нормально запускать миграции на сервере;
- нельзя безопасно чистить/строить cache на сервере;
- легко случайно затереть `.env` или `storage`;
- обновления будут ручными и более рискованными;
- при изменении схемы БД нужно руками готовить SQL-дамп или SQL-патч.

### GitHub + SSH

Плюсы:

- нормальный repeatable deploy;
- можно запускать `composer install`, `php artisan migrate --force`, `php artisan optimize`;
- проще откатываться;
- можно собирать фронт в GitHub Actions или локально.

Минусы:

- нужен SSH-доступ или хотя бы SFTP + возможность выполнить remote commands;
- нужно один раз настроить deploy key/secret;
- нужно аккуратно отделить production `.env` и `storage` от репозитория.

### GitHub Actions + FTP без SSH

Технически возможно: GitHub Actions может собрать проект и синхронизировать файлы по FTP.

Но это не полноценный Laravel CI/CD, потому что без SSH он не сможет безопасно:

- выполнить миграции;
- выполнить `storage:link`;
- очистить/пересобрать cache на сервере;
- проверить права на папки;
- сделать rollback БД.

Такой вариант можно использовать только как автоматизированную заливку файлов, а БД и `storage` все равно обслуживать руками.

## Что попросить у хостинга или у человека на удаленном компьютере

Минимум:

- PHP `8.3+` или `8.4`;
- MySQL `8` или совместимая версия;
- FTP/SFTP;
- phpMyAdmin/Adminer;
- возможность выбрать document root на `ds-art-app/public` или положить Laravel-приложение вне `public_html`;
- возможность создать symlink `public/storage -> storage/app/public`;
- права записи на `storage` и `bootstrap/cache`;
- лимиты загрузки файлов: `upload_max_filesize`, `post_max_size`, `max_execution_time`.

Если у тебя прямо сейчас есть только FTP и больше ничего, то этого недостаточно для гарантированно корректного first deploy Laravel-приложения с живой БД.

Лучше попросить дополнительно:

- SSH;
- Composer на сервере;
- PHP CLI той же версии, что и веб-PHP;
- Cron, если позже понадобится отложенная публикация по расписанию;
- автоматические бэкапы MySQL и файлов.

## Как работает база и файлы на production

Приложение работает с MySQL как с обычной живой БД. Данные будут обновляться сразу при HTTP-запросах: создание статей, изменение прав, загрузка файлов, теги, черновики и сотрудники пишутся в MySQL и `storage`.

Отдельный realtime-сервер сейчас не нужен. Здесь нет WebSocket-синхронизации между открытыми вкладками, но для текущего MVP это нормально: данные сохраняются на backend и подтягиваются при переходах/обновлении Inertia-страниц.

Критично для файлов:

- `FILESYSTEM_DISK=public`;
- сервер должен писать в `storage/app/public`;
- браузер должен видеть эти файлы через `public/storage`;
- нельзя затирать `storage/app/public` при обновлениях.
