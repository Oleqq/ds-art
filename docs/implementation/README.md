# Реализация MVP

Эта папка нужна не для описания идеи проекта, а для практического сопровождения разработки.

Здесь хранится:

- что именно мы делаем
- в каком порядке мы это делаем
- что входит в MVP
- что не входит в MVP
- какие правила нельзя нарушать по ходу реализации
- на каком этапе находится проект

Главный ориентир:

- текущий файл `docs/prototype/staff-cabinet (10).html` является целевым продуктовым состоянием MVP

Это значит:

- если сценарий существует в HTML-прототипе, он относится к MVP
- если сценария в HTML нет, мы не добавляем его автоматически
- любые расширения делаются только после отдельного решения

## Структура папки

- `STATUS.md`
  - текущий статус этапов
- `RULES.md`
  - обязательные правила реализации
- `areas/`
  - разрез по предметным областям
- `phases/`
  - пошаговые фазы реализации
- `DEPLOY_FTP.md`
  - инструкция по заливке и подъему приложения через FTP/phpMyAdmin
- `FINAL_AUDIT_AND_DEPLOY.md`
  - финальный аудит готовности, сравнение FTP-only и GitHub/SSH-деплоя, production-чек-лист

## Как пользоваться этой папкой

Перед началом новой задачи:

1. Проверить `STATUS.md`
2. Проверить соответствующую фазу в `phases/`
3. Проверить соответствующую область в `areas/`
4. Убедиться, что задача реально нужна для MVP

После завершения задачи:

1. Обновить статус этапа
2. Зафиксировать, что реализовано
3. Зафиксировать, что было сознательно отложено

## Основной принцип

Проект строится от продукта к коду, а не от технологий к технологиям.
