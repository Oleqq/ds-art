## Техническая позиция

Редактор intentionally собран не как “магический WYSIWYG-комбайн”, а как управляемый block-editor.

Причина:

- block-editor лучше ложится на логику прототипа
- им проще управлять по правам, рендерингу и будущему поиску
- один и тот же формат `blocks` можно использовать для:
  - админского редактирования
  - employee read-only рендера
- будущего поиска
- AI-слоя

## Как статьи лежат в базе данных

На этапе 5 зафиксирован важный backend-вывод: статья не должна жить только как визуальная форма на фронте.

Сейчас используется трехуровневая структура:

- `knowledge_articles` — главная запись статьи, title/slug/summary/content/blocks/cover/published
- `knowledge_article_blocks` — каждый блок статьи отдельной SQL-строкой
- `knowledge_article_assets` — каждый файл, картинка или видео отдельной SQL-строкой

Это сделано, чтобы:

- в Adminer можно было видеть содержимое статей без разбора интерфейса
- SQL-запросами можно было найти конкретный текст, файл или картинку
- будущий поиск мог работать по `plain_text`
- будущий AI-слой мог читать базу знаний структурно
- перенос на другой фронт не ломал накопленный контент

Подробное описание вынесено в `docs/implementation/areas/database-content-storage.md`.

## Как здесь используется Inertia

`Inertia` в этом этапе нужна не ради самого факта “у нас React”.

Она используется для того, чтобы:

- переход в статью происходил без полной перезагрузки страницы
- сохранение статьи обновляло серверное состояние без `F5`
- duplicate / move / delete работали как часть SPA-flow
- toolbar и редактор ощущались как приложение, а не как набор Blade-форм

То есть:

- `Laravel` остается источником данных, прав и маршрутов
- `React` отвечает за редактор и реактивность интерфейса
- `Inertia` связывает это в один поток без hard refresh
