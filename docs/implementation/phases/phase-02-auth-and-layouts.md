# Этап 02. Аутентификация, вход, роли, layouts

## Цель

Перевести starter-auth в продуктовую схему проекта: безопасный вход, разделение `admin` и `employee`, отдельные layout-потоки и backend-ограничение доступа.

## Что важно понимать

Базовый auth-слой уже частично есть в starter.  
На этом этапе мы не изобретаем аутентификацию с нуля, а адаптируем ее под реальную модель приложения.

## Результат этапа

После завершения этапа должно быть:

- пользователь может войти и выйти из системы
- `admin` попадает в свою часть приложения
- `employee` попадает в свою часть приложения
- неавторизованный пользователь не имеет доступа к защищенным экранам
- неактивный пользователь не может работать в системе
- маршруты и layout-логика разделены по ролям

## Уточнение по регистрации

Публичную регистрацию на этом этапе не делаем, если не появится отдельное бизнес-требование.

## Практический итог

- публичная регистрация отключена на уровне Fortify
- пользователь `admin` попадает в `/admin`
- пользователь `employee` попадает в `/employee`
- неактивный пользователь не проходит аутентификацию
- role/active-проверки вынесены в backend middleware
- базовый auth-каркас готов к переходу в модуль сотрудников
