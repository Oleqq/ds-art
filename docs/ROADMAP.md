# Дорожная карта проекта

## Контекст

Текущее состояние проекта:

- статический SPA-прототип в файле `staff-cabinet (3).html`

Этот HTML уже описывает не просто “базу знаний”, а внутренний корпоративный продукт:

- список сотрудников
- карточка сотрудника
- статусы сотрудников
- график работы
- заметки руководителя
- файлы сотрудника
- дерево базы знаний
- категории и статьи
- права доступа
- редактор статьи
- вложения, обложки, эмодзи, перемещения, дублирование

Главный принцип проекта:

- реализуем то, что уже есть в HTML-прототипе
- не добавляем технологии и подсистемы “на всякий случай”
- не делаем функционал ради функционала

## Базовые инженерные принципы

### 1. Минимально достаточный стек

Используем только то, что реально нужно для текущего этапа.

Сразу берем:

- `Laravel 13`
- `PHP 8.4` через `Laravel Herd`
- `MySQL`
- `Inertia.js`
- `React`
- `TypeScript`
- `Vite`

Не берем сразу:

- отдельный frontend-репозиторий
- отдельный backend API-сервис
- микросервисы
- Redux, Zustand и другой глобальный state manager без необходимости
- WebSocket/realtime без реального сценария
- Scout, Meilisearch, Horizon, Pulse, Redis, если на этапе это не нужно
- избыточные паттерны типа repository layer “по привычке”

### 2. MVP = паритет с HTML-прототипом

В этом проекте MVP не урезается до “простого CRUD”.

MVP должен покрыть всю текущую продуктовую логику, которая уже показана в HTML:

- кабинет администратора
- кабинет сотрудника
- дерево БЗ
- статьи
- права
- вложения
- действия со статьями и категориями
- поведение интерфейса, близкое к прототипу

Это не значит, что все делается в один спринт.

Это значит:

- MVP делится на этапы
- но продуктовая цель MVP остается полной паритетностью с текущим HTML

### 3. Модульный монолит

Проект строим как `modular monolith`.

Почему:

- предметная область еще формируется
- права доступа и редактор тесно связаны
- быстрее разработка, тестирование и деплой
- меньше лишней инфраструктурной сложности

### Подключаемые по необходимости технологии

- `Redis`
  - подключаем, когда реально появляются очереди, кэширование или тяжелый поиск
- `Laravel Horizon`
  - подключаем только когда появляются очереди, которые надо мониторить
- `Laravel Pulse`
  - подключаем на этапе эксплуатации или при первых проблемах наблюдаемости
- `Laravel Scout`
  - подключаем, когда обычного поиска по БД перестает хватать
- `Meilisearch`
  - подключаем только если действительно нужен быстрый полнотекстовый поиск и ранжирование
- `Tiptap`
  - подключаем, если становится ясно, что редактор из прототипа нельзя адекватно поддерживать на простом rich-text уровне

## Как собираем фронтенд

## Общий подход

Фронтенд не выносим в отдельный SPA-проект.

Используем:

- `Laravel + Inertia + React + TypeScript + Vite`

Это значит:

- роуты и доступ контролирует Laravel
- страницы отдаются как Inertia responses
- React отвечает за интерактивность, локальное состояние и сложные UI-элементы
- отдельный frontend backend-for-frontend не нужен

## Схема работы

### Навигация

- `Laravel routes` определяют URL и доступ
- контроллеры возвращают `Inertia::render(...)`
- React-страницы живут внутри `resources/js`

### Сборка

- в dev-режиме фронтенд собирается через `Vite`
- в production используется `vite build`
- отдельного webpack, отдельного dev-server приложения и отдельного frontend deploy не делаем

### Структура фронтенда

Рекомендуемая структура:

- `resources/js/pages`
- `resources/js/features`
- `resources/js/components`
- `resources/js/layouts`
- `resources/js/lib`

По фичам:

- `features/employees`
- `features/knowledge-base`
- `features/access`
- `features/files`
- `features/editor`

### Состояние

На старте:

- локальный React state
- form state внутри страниц и компонентов
- page props от Inertia

Не добавляем сразу:

- Redux
- Zustand
- React Query

Добавляем только если появится реальная сложность:

- общий кэш большого числа async-запросов
- сложная синхронизация нескольких экранов
- тяжелое клиентское состояние, которое невозможно держать локально

### Стили

Чтобы не тащить лишний слой абстракции, на старте не берем Tailwind как обязательную основу.

Рекомендуемый подход:

- `CSS variables`
- обычные `.css` или `.scss` файлы
- переиспользуемые UI-компоненты

Почему так:

- текущий прототип уже мыслит себя как обычная дизайн-система на CSS-переменных
- так проще переносить существующую верстку
- меньше зависимости от utility-first слоя, если он не нужен

Для MVP дополнительно фиксируем:

- интерфейс держим в светлой теме, как в HTML-прототипе
- dark mode не считаем частью текущего MVP
- возврат к theme switcher возможен позже как отдельная задача, но не во вред паритету с прототипом

### Frontend API

На старте не делаем “API ради API”.

Используем:

- обычные web routes для страниц
- JSON endpoints только для async-действий

Примеры async-действий:

- reorder дерева
- загрузка файлов
- сохранение блоков статьи
- поиск
- массовые изменения доступа

## Как строим бэкенд

## Общий подход

Бэкенд строим как Laravel-монолит с четким разделением по модулям.

Не строим:

- отдельный REST backend для всего приложения
- лишний repository/service zoo
- CQRS/Event Sourcing без реальной необходимости

## Правила проектирования бэкенда

### Контроллеры

Контроллеры должны быть тонкими.

Они:

- принимают запрос
- валидируют через Form Request
- вызывают action/service
- возвращают Inertia response или JSON

### Бизнес-логика

Бизнес-логика живет не в контроллерах и не в моделях “как попало”, а в явных прикладных сценариях:

- `CreateEmployee`
- `UpdateEmployee`
- `MoveArticle`
- `SaveArticleAccess`
- `CreateKnowledgeCategory`

### Авторизация

Авторизация должна идти через:

- `Policies`
- `Gates`
- явные проверки на уровне сценария

Нельзя полагаться только на скрытие кнопок на фронте.

### Транзакции

Транзакции используем там, где есть связные изменения:

- создание статьи + привязка к категории
- изменение прав
- перемещение статьи
- удаление категории с дочерними элементами

### Файлы

Файлы должны храниться через `Laravel Storage`.

В БД храним:

- путь
- оригинальное имя
- mime type
- размер
- владельца
- тип сущности, к которой файл прикреплен

Нельзя хранить production-файлы в формате base64 в БД, как это сейчас делает прототип.

## Этап 9. AI-агент как “cherry on top”

Цель:

- добавить интеллектуальный слой поверх уже работающего продукта

Этот этап не входит в базовую реализацию MVP.

Он начинается только после того, как:

- данные нормально структурированы
- права доступа работают корректно
- поиск и индексация уже дают стабильный результат

### Что должен уметь агент

- отвечать по статьям базы знаний
- помогать быстро находить нужный контент
- анализировать связанные данные по сотрудникам, статьям, файлам и другим внутренним сущностям
- помогать с навигацией по системе

### Как его правильно делать

Не делать “агента, который читает все подряд”.

Нужно делать:

- ограниченный слой доступа к данным
- RAG/поисковый слой по статьям
- отдельные policy-проверки на каждый запрос
- логирование обращений

Принцип:

- агент видит только то, что имеет право видеть конкретный пользователь

### Что нужно заранее для этого этапа

- чистая схема БД
- нормальные связи между сущностями
- доступный индекс статей
- корректный ACL
- понятные entities

## Полезные улучшения после MVP

Это не обязательная часть базовой реализации, но это типичные полезные функции для современных внутренних knowledge-base / staff-cabinet систем:

- поддержка светлой и темной темы для всего приложения
- recent activity по сотрудникам, статьям и файлам
- pinned / избранные статьи и разделы
- recently viewed для быстрого возврата
- шаблоны статей для типовых материалов
- история изменений карточки сотрудника
- soft archive вместо жесткого удаления там, где это полезно
- напоминания о незаполненных профилях или устаревших статьях

Эти идеи не добавляем автоматически.

Правило:

- сначала закрываем полный паритет с текущим HTML-прототипом
- потом принимаем отдельное решение, что из этого реально нужно продукту

Для темы фиксируем отдельно:

- в MVP держим светлую тему как базовую, чтобы сохранить точный визуальный паритет с текущим прототипом
- full `light/dark` support планируем как отдельный post-MVP слой
- theme switcher делаем только после того, как будут готовы основные экраны, роли и база знаний
